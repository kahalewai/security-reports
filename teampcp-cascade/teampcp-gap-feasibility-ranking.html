TeamPCP Gap Solutions — Build Feasibility Ranking

10 unsolved AppSec gaps ranked from easiest to hardest to build and implement. Based on the TeamPCP Supply Chain Campaign analysis (Feb–Mar 2026).

Buildable now — weeks to low months
Moderate — scoped but needs coordination
Hard — multi-year, cross-industry effort
# Difficulty Problem Solution Tech Stack Delivery Channel
Buildable Now
1 Easy Gap 2.2 No Detection for C2 Domain Rotation in CI/CD Attackers used a fresh typosquatted domain per vendor wave. Blocking one IOC gave zero protection against the next. Action authors publish a network manifest declaring which domains their action contacts. Any undeclared egress from a CI runner triggers an alert. Pair with DNS-permutation monitoring for vendor domains.
YAML schema eBPF / netfilter dnstwist GitHub Actions metadata
CI/CD platform feature GitHub Actions extension Open-source CLI tool
2 Easy Gap 3.5 (partial) No Credential Cascade Visibility No tool maps which 3rd-party actions can access which secrets, or which downstream systems those secrets unlock. The entire chain from Node ③→⑦ was invisible beforehand. Static analysis tool that parses CI/CD pipeline configs (Actions YAML, GitLab CI) and produces a graph: action → secrets accessed → downstream systems reachable. Visualizes blast radius before incidents.
YAML parser Graph DB / D3.js GitHub API Python / Go
Open-source CLI tool GitHub App SaaS dashboard
3 Easy Gap 3.3 No Security Model for IDE Extension Supply Chains Trojanized Checkmarx extensions on OpenVSX ran with full user permissions. No permission model, no signing standard, no cross-registry integrity checks. Permission declaration model (like mobile apps) for extensions. Certificate-pinned signing. Cross-registry diff service that alerts when the same extension diverges between Marketplace and OpenVSX.
VS Code Extension API Code signing (X.509) Registry APIs Diff engine
IDE platform feature Registry policy change Open-source monitor
Moderate — Scoped but Needs Adoption
4 Moderate Gap 1.2 No Agent Identity or Attribution Framework hackerbot-claw opened PRs across thousands of repos with no way for maintainers to identify it as an autonomous agent vs. a human researcher. Voluntary agent identity header in PRs (like User-Agent for crawlers). Platform-enforced bot labeling. Behavioral heuristics detecting agent-like patterns: rapid cross-repo PRs, uniform payloads, inhuman timing.
Platform APIs Anomaly detection (ML) Metadata schema Rate limiting
GitHub / GitLab policy Open standard (RFC) Platform-native detection
5 Moderate Gap 3.4 No Detection for AI-Generated Malware CanisterWorm was "vibe-coded" — clean, well-commented, structurally normal. Existing scanners look for obfuscation patterns that AI-generated malware doesn't have. Registry-level sandboxed execution of install hooks. Monitor for credential enumeration, exfiltration, and persistence creation at runtime — catching intent regardless of code style.
Container sandboxing Syscall tracing (strace/seccomp) Behavioral analysis Registry integration
PyPI / npm infrastructure Open-source sandbox tool Registry pre-publish gate
6 Moderate Gap 3.1 (partial) No Trust Framework for AI Infrastructure Packages LiteLLM holds every LLM API key by design — present in 36% of cloud environments. No framework distinguishes its risk from a date-formatting library with the same download count. Privilege classification taxonomy: packages self-declare credential categories they require. "Blast radius score" combining downloads × privilege level. Enhanced registry review for high-privilege packages.
Registry metadata API Scoring engine Manifest schema Static analysis
PyPI / npm feature OpenSSF scorecard plugin SCA vendor integration
7 Moderate Gap 3.5 (full) No Cross-Vendor Credential Cascade Prevention Credentials from Trivy's CI → npm → KICS → LiteLLM. Each compromised env yielded keys to the next. No isolation between pipeline stages consuming different vendors' tools. Pipeline stage isolation: each stage (scan, build, publish) gets only its own credentials. Cross-ecosystem revocation protocol: automated notification + rotation when a vendor tool is compromised.
OPA / policy engine OIDC scoped tokens Webhook orchestration Secrets managers
CI/CD platform architecture OpenSSF working group Industry protocol (FIRST)
Hard — Multi-Year / Cross-Industry
8 Hard Gap 1.1 No Framework for Agent Attack Speed and Scale hackerbot-claw scanned 47K+ repos, classified 9 vuln classes, generated custom exploits, and achieved code execution — all in 37 hours. No threat model accounts for this speed. Extend STRIDE / MITRE ATT&CK with an "autonomous agent" adversary profile: continuous scanning, adaptive exploits, parallelized execution, sub-minute pivots. Redefine remediation SLAs accordingly.
MITRE ATT&CK framework Threat modeling tools Standards bodies
MITRE / OWASP standard Threat modeling extension Training / certification
9 Hard Gap 2.1 No Detection or Takedown for Blockchain-Based C2 CanisterWorm used an ICP canister as a dead-drop C2. No registrar, no single hosting provider, no takedown point. EDR/NDR tools can't distinguish it from legitimate blockchain traffic. Endpoint behavioral detection (polling → payload → execute pattern). Network heuristics for ICP traffic from CI runners. Industry consortium for decentralized network abuse reporting. Registry scanners for canister URLs.
EDR/NDR integration TLS fingerprinting Blockchain APIs Legal frameworks
EDR vendor feature OpenSSF / FIRST consortium Law enforcement coordination
10 Hard Gap 3.2 No SBOM for AI Agent Dependency Chains A developer never installed LiteLLM directly — it arrived via Cursor → MCP plugin → transitive dep. This chain is invisible to CycloneDX/SPDX and every existing SBOM tool. Extend CycloneDX/SPDX with AI artifact types (MCP servers, agent skills, IDE extensions). Runtime dependency tracker monitoring actual installs during agent operation. Dependency transparency mandate for MCP servers.
CycloneDX / SPDX Runtime agent (eBPF) MCP protocol IDE plugin APIs
Standards body (OWASP) Open-source runtime agent IDE-level SBOM generator
11 Hard Gap 1.3 No Defense Model for Agent-vs-Agent CI/CD Warfare hackerbot-claw tried to poison a CLAUDE.md file to trick a Claude-based CI defender into injecting malicious code. First documented AI-vs-AI attack in a pipeline. Only one known instance. Prompt injection resistance standards for CI agents. Tool allowlisting per pipeline stage. Separation of AI code generation from AI code approval. Behavioral anomaly monitoring for agent pipeline activity.
Prompt hardening suites Policy-as-code (OPA) Agent sandboxing LLM eval frameworks
AI vendor certification CI/CD platform feature New security discipline
